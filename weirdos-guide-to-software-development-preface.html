Weirdo's Guide to Software Development - Preface

Date Fri 22 January 2016 Tags Software Development / Tutorials

Hi there, welcome to my little introduction to software design and development. This is going to be a series of tutorials that will focus on a language agnostic approach to designing and developing good software, hopefully.

Please bear in mind that this series is not in any way meant to replace a full fledged educational degree. Instead it aims to give people an idea on what they could do or rather what is usually done to make good software.

Motivation

I'm not sure how it is for most new graduates/developers but I find that more often than not people tend to consider software development to consist of seeing a problem description and then jumping straight into coding and mucking around until trial and error produces a decent solution. This is a pretty bad approach, one that I have been guilty of as well. So I decided to start this. An objective approach following the principles of a Software Development Life Cycle (SDLC). Documenting things like this in a blog will also force me to actually follow a good approach and if someone else happens to find this series useful, even better!

Things To Keep In Mind

  1. Since this is technically just me working on this and not a team or anything I'll be using the simple Waterfall model for this process.
  2. In the interest of brevity and more importantly my own sanity, I won't be making a full Software Requirement Specification (SRS) or a Product Requirements Document (PRD). Mostly the documentation for this series will consist of a few short snippets, lists and diagrams that will be here in the blog itself.
  3. While this guide will use Python, it is not intended to be a Python coding/programming tutorial and will usually focus more on good design practices. If I feel like its a good idea, I might make some tutorials about coding or programming later on.
  4. Python is being used here solely for the speed of development and readability, if in the future, I happen to have some readers and there is a demand for some other language to be used in a project, I will be happy to oblige, as long as its not something extreme like BrainFuck.
  5. If at any point you see any problems, something I've said wrong or spelled incorrectly or written something incomprehensible then please leave a comment about it. I'd be more than happy to improve the tutorial with your feedback, not to mention my own approach.

Now that that's out of the way, lets begin with the introduction
